# Design Producer Protocol

The harness can delegate style production to an external command:

```bash
HARNESS_SCRIPT="$SKILL_ROOT/scripts/harness.py"
python3 "$HARNESS_SCRIPT" style propose \
  --producer command \
  --producer-command "./scripts/design-skill-producer" \
  --base workspace/products/codefox/codefox/brand.lock.yaml \
  --brief workspace/products/codefox/codefox/brief.md \
  --source workspace/products/codefox/codefox/references/ \
  --out workspace/products/codefox/codefox/proposals/codefox.lock.yaml
```

The command receives JSON on stdin:

```json
{
  "version": "1.2.0",
  "base_brand": {
    "version": "1.1.0",
    "provider": {},
    "global": {},
    "alias": {}
  },
  "brief": "Brand brief markdown content",
  "references": ["workspace/products/codefox/codefox/references/main_visual.png"],
  "output_contract": "complete brand.lock YAML or JSON on stdout"
}
```

The command must write a complete `brand.lock` YAML or JSON document to stdout. It should not write explanations to stdout. Send diagnostics to stderr.

The harness will:

1. Parse stdout as YAML/JSON.
2. Set or confirm the requested version.
3. Validate the complete brand lock schema.
4. Validate token `$value` / `$type` structure.
5. Validate alias references.
6. Write the proposal to `workspace/products/codefox/codefox/proposals/*.lock.yaml`.

The producer must not:

- Read or write `.env`.
- Put API keys in any output.
- Put campaign content in brand tokens.
- Publish or render images directly.
- Modify the official brand lock path unless the user explicitly requested promotion.

Recommended output metadata can be stored in a sidecar generated by the producer, for example:

```text
workspace/products/codefox/codefox/proposals/codefox.provenance.json
```

The harness does not require provenance yet, but it is useful for review.
